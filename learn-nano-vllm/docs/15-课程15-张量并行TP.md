# 课程 15：张量并行（Tensor Parallelism）

> **学习目标**：把张量并行放回 nano-vllm 的真实执行链路里理解；分清 `tp_rank / tp_size / 列并行 / 行并行 / 本地计算 / 通信` 这些最容易混的概念；看懂 `ColumnParallelLinear`、`RowParallelLinear`、`QKVParallelLinear`、`MergedColumnParallelLinear`、`VocabParallelEmbedding` 和 `ParallelLMHead` 的当前实现；掌握“为什么 Transformer 里总是先列并行，再局部算，再行并行 + 通信”这条主线。

---

## 零、先讲人话版

如果你第一次读张量并行，很容易把它和“多请求并发”“多卡各跑一部分用户请求”混在一起。先把一句话记死：

> **张量并行不是把请求分给多张卡，而是把同一个线性层的同一个大矩阵，拆给多张 GPU 一起算。**

也就是说，在 TP（Tensor Parallelism）里：

- 同一条 `Sequence`
- 同一个 batch
- 同一层 Transformer
- 所有 rank 都会参与

区别只在于：

- 每个 rank 手里拿的是**不同的权重分片**
- 因此每个 rank 算出来的也是**不同的局部结果**
- 什么时候需要通信，取决于这些局部结果能不能直接继续往下算

先记住 nano-vllm 里一层 Transformer 的主线：

```text
输入 hidden_states（所有 rank 一样）
  -> QKVParallelLinear：按头切开，每个 rank 只算自己那部分 Q/K/V
  -> Attention：每个 rank 只对自己的头做局部注意力
  -> o_proj(RowParallelLinear)：把各 rank 的局部结果求和，恢复完整 hidden_states
  -> gate_up_proj(MergedColumnParallelLinear)：再切开中间维度
  -> 激活 + 逐元素乘：仍然是局部算
  -> down_proj(RowParallelLinear)：再一次求和，恢复完整 hidden_states
```

如果把它压成一句更短的话，就是：

> **先把“大输出维度”拆开各算各的，等必须合起来时，再用通信恢复完整结果。**

这里先提前澄清 5 个最容易混淆的点：

1. **TP 的切分对象是“层里的张量”，不是请求。**
   所以 TP 和课程 12/13/14 里的 waiting、running、prefill、decode 不是同一维度的概念。

2. **ColumnParallel 的输出通常是“shape 不完整，但值是完整的本地分片”。**
   例如每张卡只拿到隐藏维度的前半/后半。

3. **RowParallel 的输出通常是“shape 完整，但值只是部分和”。**
   所以必须 `all_reduce(sum)` 才能得到真正完整结果。

4. **有些本地操作不需要通信。**
   比如 `split`、`view`、RoPE、每个头独立 attention、SwiGLU 里的逐元素乘，这些都能在各 rank 自己的 shard 上完成。

5. **LM Head 在当前 nano-vllm 里不是所有 rank 都拿完整 logits。**
   当前实现是各 rank 先各算一部分词表 logits，再 `gather` 到 rank 0，由 rank 0 采样。

如果你读到后面晕了，随时退回来，只问自己两件事：

- 这一层现在切的是**输出维度**还是**输入维度**？
- 当前每个 rank 手里的结果，是“局部完整分片”，还是“完整形状的部分和”？

## 一、先把边界讲清楚：TP 到底在并行什么

### 1.1 TP 不是“多卡各处理一个请求”

在 nano-vllm 里，调度器决定的是：

- 这轮有哪些 `Sequence`
- 这轮是 prefill 还是 decode
- 每条序列这轮算多少 token

这些属于**请求级 / 序列级调度**。

而张量并行决定的是：

- 一个线性层的权重怎么分到多张 GPU
- 多张 GPU 算出的局部结果何时需要通信

这些属于**单层内部的张量级并行**。

所以你可以把二者理解成：

| 层次 | 回答的问题 |
|------|-----------|
| 调度层 | 这轮谁上 GPU？ |
| 模型执行层 | 这些 token 怎么变成前向输入？ |
| 张量并行层 | 一个层内部的权重和计算怎么分给多张 GPU？ |

### 1.2 `tp_rank` 和 `tp_size` 是什么

源码路径：

- `nanovllm/layers/linear.py`
- `nanovllm/layers/embed_head.py`
- `nanovllm/models/qwen3.py`

当前实现里，每个 TP 进程都会通过：

```python
self.tp_rank = dist.get_rank()
self.tp_size = dist.get_world_size()
```

拿到：

- `tp_rank`：我是第几张卡 / 第几个 TP 进程
- `tp_size`：总共参与 TP 的卡数

例如 `tp_size = 2` 时：

- `tp_rank = 0` 负责第 0 份
- `tp_rank = 1` 负责第 1 份

### 1.3 nano-vllm 的多进程布局

结合课程 14 一起看会更清楚。

当前 `LLMEngine` 会：

1. 用 `spawn` 启动 `tp_size - 1` 个子进程；
2. 每个子进程各自构造一个 `ModelRunner(config, rank=i, event=event)`；
3. 主进程里再构造 `rank=0` 的 `ModelRunner`；
4. 所有 rank 用 NCCL 建立进程组；
5. forward 阶段由 NCCL 做张量通信。

也就是说：

- **SharedMemory / Event**：负责“方法调用和序列元信息”同步
- **NCCL**：负责模型前向里的 `all_reduce` / `gather`

这两个通道不要混。

## 二、Transformer 里 TP 的主线为什么总是“先列后行”

如果只背概念，很容易背成散的。真正该记的是 Transformer 里最常见的这条配对模式：

```text
完整输入
  -> ColumnParallelLinear
  -> 一串不需要跨 shard 混合的本地操作
  -> RowParallelLinear
  -> 恢复完整输出
```

为什么总是这样？

因为：

1. **先按输出维度切开**，每张卡都能独立算出自己那一片特征；
2. 中间很多操作其实只在“各自那一片特征”内部进行，不需要别人的数据；
3. 等到下一层真的需要完整 hidden state 时，再用 RowParallel + `all_reduce` 把结果合起来。

对应到 Qwen3：

### 2.1 Attention 路径

```text
hidden_states（完整）
  -> qkv_proj：列并行，按头切
  -> q/k/v reshape + rope + attention：本地算
  -> o_proj：行并行 + all_reduce
  -> attention 输出恢复完整
```

### 2.2 FFN 路径

```text
hidden_states（完整）
  -> gate_up_proj：列并行，按中间维度切
  -> silu + mul：本地算
  -> down_proj：行并行 + all_reduce
  -> FFN 输出恢复完整
```

把这条主线吃透了，后面的 `QKVParallelLinear`、`MergedColumnParallelLinear` 都只是这个模式的具体变体。

## 三、ColumnParallelLinear：先拆输出维度，各算各的

源码路径：`nanovllm/layers/linear.py`

### 3.1 一句话先说透

`ColumnParallelLinear` 的本质是：

> **每个 rank 都看到完整输入，但只负责输出维度中的一部分。**

### 3.2 数学上的“列切分”和代码里的 `tp_dim=0` 为什么不矛盾

这是最容易把人绕晕的点。

数学上通常写：

```text
Y = XW
X: [M, K]
W: [K, N]
Y: [M, N]
```

如果按“输出维度 `N`”切分，就会说：

```text
W = [W0, W1, ..., W_{tp-1}]
```

也就是**按列切分**数学上的 `W`。

但在 PyTorch 的 `F.linear(x, weight)` 里，`weight` 的真实存储形状是：

```text
[output_size, input_size]
```

也就是：

```text
[N, K]
```

因此，数学上“按输出列切分”，落到 PyTorch 参数张量上，反而对应：

```python
tp_dim = 0
```

也就是沿着 `weight` 的第 0 维切。

这是 `ColumnParallelLinear` 最容易误读的地方，面试时如果你能主动讲清这层“数学矩阵”和“PyTorch 参数布局”的错位，会很加分。

### 3.3 当前实现

```python
class ColumnParallelLinear(LinearBase):
    def __init__(self, input_size, output_size, bias=False):
        tp_size = dist.get_world_size()
        super().__init__(input_size, divide(output_size, tp_size), bias, 0)

    def weight_loader(self, param, loaded_weight):
        param_data = param.data
        shard_size = param_data.size(self.tp_dim)
        start_idx = self.tp_rank * shard_size
        loaded_weight = loaded_weight.narrow(self.tp_dim, start_idx, shard_size)
        param_data.copy_(loaded_weight)

    def forward(self, x):
        return F.linear(x, self.weight, self.bias)
```

读法很直接：

- 每张卡保留 `output_size / tp_size` 个输出通道；
- 输入 `x` 不切；
- 前向直接做本地 `linear`；
- 不做通信。

### 3.4 它的输入和输出长什么样

假设：

```text
input_size = 4096
output_size = 8192
tp_size = 2
```

则每个 rank 的权重形状是：

```text
weight: [4096, 4096]   <- 注意这里指 PyTorch Linear 的 [out, in] 语义
```

如果输入：

```text
x: [M, 4096]
```

那么每个 rank 的输出都是：

```text
y_local: [M, 4096]
```

但要注意，这个 `[M, 4096]` 不是完整输出，而只是完整 `[M, 8192]` 的一半。

所以 ColumnParallel 的输出特征是：

> **shape 不完整，但本 rank 这部分值已经是最终正确值。**

### 3.5 为什么 ColumnParallel 通常不需要立刻通信

因为后面很多操作，本来就可以只在自己这部分特征上做：

- attention 里每个 rank 只管自己的头
- FFN 里每个 rank 只管自己的中间维度
- `split`、`view`、RoPE、逐元素激活，都不要求“先把别人的 shard 拼过来”

所以通信可以推迟。

## 四、RowParallelLinear：输入已经分片，输出先是“部分和”

源码路径：`nanovllm/layers/linear.py`

### 4.1 一句话先说透

`RowParallelLinear` 的本质是：

> **每个 rank 只吃输入中的一部分，但会产出完整输出形状的“部分和”，最后再通过 `all_reduce(sum)` 合成真正完整输出。**

### 4.2 当前实现

```python
class RowParallelLinear(LinearBase):
    def __init__(self, input_size, output_size, bias=False):
        tp_size = dist.get_world_size()
        super().__init__(divide(input_size, tp_size), output_size, bias, 1)

    def weight_loader(self, param, loaded_weight):
        param_data = param.data
        if param_data.ndim == 1:
            param_data.copy_(loaded_weight)
            return
        shard_size = param_data.size(self.tp_dim)
        start_idx = self.tp_rank * shard_size
        loaded_weight = loaded_weight.narrow(self.tp_dim, start_idx, shard_size)
        param_data.copy_(loaded_weight)

    def forward(self, x):
        y = F.linear(x, self.weight, self.bias if self.tp_rank == 0 else None)
        if self.tp_size > 1:
            dist.all_reduce(y)
        return y
```

这里要盯住两件事：

1. `input_size` 被按 `tp_size` 除掉了
2. `forward()` 里有 `dist.all_reduce(y)`

### 4.3 为什么 RowParallel 的输出 shape 是完整的，但值还不完整

还是用矩阵乘法看最直观。

假设完整输入是：

```text
X = [X0, X1]
```

完整权重按输入维度切开：

```text
W = [W0;
     W1]
```

那么：

```text
Y = XW = X0W0 + X1W1
```

于是：

- rank 0 算 `X0W0`
- rank 1 算 `X1W1`

这两个张量的 shape 都已经是：

```text
[M, output_size]
```

但它们只是最终结果的两部分和，必须再做：

```text
Y = Y0 + Y1
```

所以 RowParallel 的输出特征是：

> **shape 完整，但值还只是 partial sum。**

### 4.4 为什么 bias 只在 rank 0 加

当前实现：

```python
self.bias if self.tp_rank == 0 else None
```

原因非常经典。

如果每个 rank 都加一遍 bias，那么 `all_reduce(sum)` 之后，bias 会被重复加 `tp_size` 次。

正确做法是：

- 只让一个 rank 加 bias
- 然后随 `all_reduce(sum)` 一起汇总

这样最终只会出现一次 bias。

### 4.5 为什么 RowParallel 是 ColumnParallel 的天然搭档

因为上一个 ColumnParallel 输出的，正好就是“按特征维度切开的分片”；
而 RowParallel 需要的，正好就是“输入按该维度分片”。

所以：

```text
ColumnParallel 的输出分片
  -> 可以直接喂给
RowParallel 的输入分片
```

这就是 Transformer 里总是“先列后行”的根本原因。

## 五、Attention 里 TP 到底怎么流

源码路径：`nanovllm/models/qwen3.py`

### 5.1 `Qwen3Attention` 先把头数按 rank 分掉

当前实现一上来就做了：

```python
tp_size = dist.get_world_size()
self.total_num_heads = num_heads
assert self.total_num_heads % tp_size == 0
self.num_heads = self.total_num_heads // tp_size

self.total_num_kv_heads = num_kv_heads
assert self.total_num_kv_heads % tp_size == 0
self.num_kv_heads = self.total_num_kv_heads // tp_size
```

这说明：

- query 头会按 rank 平均分
- KV 头也会按 rank 平均分

所以 TP 不是“一个 attention 层拆半”，更准确地说是：

> **每个 rank 负责一部分 attention heads。**

### 5.2 `qkv_proj` 为什么是列并行

当前代码：

```python
self.qkv_proj = QKVParallelLinear(
    hidden_size,
    self.head_dim,
    self.total_num_heads,
    self.total_num_kv_heads,
    bias=qkv_bias,
)
```

它本质上还是 ColumnParallel：

- 输入 hidden state 完整共享给所有 rank
- 每个 rank 只算自己那部分头对应的 Q/K/V

前向之后：

```python
qkv = self.qkv_proj(hidden_states)
q, k, v = qkv.split([self.q_size, self.kv_size, self.kv_size], dim=-1)
```

这里的：

- `self.q_size = self.num_heads * self.head_dim`
- `self.kv_size = self.num_kv_heads * self.head_dim`

注意这些已经是**每个 rank 本地的头数**，不是全局总头数。

### 5.3 为什么 attention 本体不需要跨 rank 通信

因为每个 rank 已经只持有自己那部分头：

```python
q = q.view(-1, self.num_heads, self.head_dim)
k = k.view(-1, self.num_kv_heads, self.head_dim)
v = v.view(-1, self.num_kv_heads, self.head_dim)
...
o = self.attn(q, k, v)
```

从 kernel 角度看，每个头的注意力本来就是独立的。

所以：

- 头内 softmax 是本地的
- 本 rank 的 Q 只和本 rank 的 K/V 打交道
- attention 输出 `o` 也是本 rank 本地头对应的输出

这一段完全可以本地完成。

### 5.4 `o_proj` 为什么必须是行并行

attention 输出经过：

```python
output = self.o_proj(o.flatten(1, -1))
```

此时 `o.flatten(1, -1)` 只包含本 rank 的头拼起来的那部分隐藏维度。

所以：

- 它天然就是一个分片输入
- 正适合 `RowParallelLinear`
- 各 rank 得到完整形状的 partial sum
- 然后 `all_reduce` 恢复完整 hidden state

这就是 attention 路径里那次通信。

### 5.5 Attention 路径的完整一句话总结

> **QKV 投影按头切开，各 rank 本地完成自己的头注意力，最后用 `o_proj(RowParallel)` + `all_reduce` 把所有头的贡献合回完整 hidden state。**

## 六、FFN 里 TP 又是怎么流的

### 6.1 `gate_up_proj` 为什么是列并行

当前实现：

```python
self.gate_up_proj = MergedColumnParallelLinear(
    hidden_size,
    [intermediate_size] * 2,
    bias=False,
)
```

FFN 第一层的任务，是把：

```text
[M, hidden_size]
```

映射到更宽的中间维度：

```text
[M, intermediate_size]
```

这类“把输出维度做大”的层，非常适合先按输出维度切开，每个 rank 只负责中间维度的一部分。

### 6.2 为什么激活函数和逐元素乘法都不需要通信

当前实现：

```python
gate_up = self.gate_up_proj(x)
x = self.act_fn(gate_up)
x = self.down_proj(x)
```

`SiluAndMul()` 的本质，是把本 rank 的 `gate` 和 `up` 分片做：

- `SiLU`
- 再逐元素乘

这些操作都只在本 rank 自己那部分中间维度内发生，不需要别人那部分数据。

这是一条非常重要的经验：

> **只要一个算子不需要跨 shard 混合信息，就可以继续推迟通信。**

### 6.3 `down_proj` 为什么又回到行并行

FFN 第二层要把中间维度压回隐藏维度：

```text
[M, intermediate_size / tp_size]
  -> [M, hidden_size]
```

每个 rank 只拿到中间维度的一部分，所以这正是 RowParallelLinear 的典型输入。

于是：

- 各 rank 本地做 `linear`
- 得到完整 hidden shape 的 partial sum
- `all_reduce` 后恢复完整 hidden state

这就是 FFN 路径里的第二次通信。

### 6.4 FFN 路径的一句话总结

> **FFN 也是先列并行把中间维度拆开，本地做激活和逐元素乘，再用 `down_proj(RowParallel)` + `all_reduce` 合回完整 hidden state。**

## 七、QKVParallelLinear：本质还是列并行，只是装得更紧

源码路径：`nanovllm/layers/linear.py`

### 7.1 为什么要有它

理论上可以用三个独立线性层：

- `q_proj`
- `k_proj`
- `v_proj`

但这样会多出三次 matmul / 更多 launch 开销。

所以当前实现把三者打包成一个大线性层：

```text
[Q | K | V]
```

一次性算出来。

### 7.2 当前构造

```python
class QKVParallelLinear(ColumnParallelLinear):
    def __init__(self, hidden_size, head_size, total_num_heads,
                 total_num_kv_heads=None, bias=False):
        tp_size = dist.get_world_size()
        total_num_kv_heads = total_num_kv_heads or total_num_heads
        self.head_size = head_size
        self.num_heads = divide(total_num_heads, tp_size)
        self.num_kv_heads = divide(total_num_kv_heads, tp_size)
        output_size = (total_num_heads + 2 * total_num_kv_heads) * self.head_size
        super().__init__(hidden_size, output_size, bias)
```

关键点：

- `output_size` 用的是**全局总头数**
- 但每个 rank 真正持有的参数，仍然会在 `ColumnParallelLinear` 里自动切成 `1 / tp_size`

### 7.3 当前 `weight_loader` 的真实逻辑

```python
def weight_loader(self, param, loaded_weight, loaded_shard_id: str):
    assert loaded_shard_id in ["q", "k", "v"]
    if loaded_shard_id == "q":
        shard_size = self.num_heads * self.head_size
        shard_offset = 0
    elif loaded_shard_id == "k":
        shard_size = self.num_kv_heads * self.head_size
        shard_offset = self.num_heads * self.head_size
    else:
        shard_size = self.num_kv_heads * self.head_size
        shard_offset = self.num_heads * self.head_size + self.num_kv_heads * self.head_size
    param_data = param_data.narrow(self.tp_dim, shard_offset, shard_size)
    loaded_weight = loaded_weight.chunk(self.tp_size, self.tp_dim)[self.tp_rank]
    param_data.copy_(loaded_weight)
```

注意这里和很多教学化写法不一样，当前 nano-vllm 实现是：

1. 先根据 `loaded_shard_id in {"q","k","v"}` 确定这次加载的是哪一段；
2. 在本 rank 的 packed 参数里找到该段偏移；
3. 再把原始权重按 TP 维度 `chunk`；
4. 只拷贝本 rank 这一块。

### 7.4 为什么 GQA 会限制 `tp_size`

例如：

```text
total_num_heads = 32
total_num_kv_heads = 4
```

如果：

```text
tp_size = 2
```

则：

- 每 rank 16 个 Q 头
- 每 rank 2 个 KV 头

没问题。

但如果：

```text
tp_size = 8
```

则 `4 / 8` 不整除，KV 头没法均匀分。

所以当前实现要求：

```python
assert total_num_heads % tp_size == 0
assert total_num_kv_heads % tp_size == 0
```

面试时别只说“Q 头要整除”，KV 头也必须整除，这点很关键。

## 八、MergedColumnParallelLinear：把两次列并行压成一次

源码路径：`nanovllm/layers/linear.py`

### 8.1 为什么要合并

FFN 里经常有两层同输入的列并行线性层一起出现，比如：

- `gate_proj`
- `up_proj`

它们输入相同，输出维度也相同。

与其：

- 各做一次 matmul

不如：

- 先把权重拼起来
- 一次 matmul 同时算出 `gate` 和 `up`

### 8.2 当前实现

```python
class MergedColumnParallelLinear(ColumnParallelLinear):
    def __init__(self, input_size, output_sizes, bias=False):
        self.output_sizes = output_sizes
        super().__init__(input_size, sum(output_sizes), bias)
```

在 Qwen3 里：

```python
output_sizes = [intermediate_size, intermediate_size]
```

也就是：

```text
[gate | up]
```

合成一个更宽的列并行层。

### 8.3 当前 `weight_loader` 的真实逻辑

```python
def weight_loader(self, param, loaded_weight, loaded_shard_id: int):
    param_data = param.data
    shard_offset = sum(self.output_sizes[:loaded_shard_id]) // self.tp_size
    shard_size = self.output_sizes[loaded_shard_id] // self.tp_size
    param_data = param_data.narrow(self.tp_dim, shard_offset, shard_size)
    loaded_weight = loaded_weight.chunk(self.tp_size, self.tp_dim)[self.tp_rank]
    param_data.copy_(loaded_weight)
```

这里的 `loaded_shard_id` 不是字符串，而是：

- `0` 表示 `gate_proj`
- `1` 表示 `up_proj`

这个映射来自 `Qwen3ForCausalLM.packed_modules_mapping`：

```python
"gate_proj": ("gate_up_proj", 0),
"up_proj": ("gate_up_proj", 1),
```

也就是说，加载器看到 HuggingFace 权重名里有 `gate_proj` / `up_proj` 时，会把它们导向同一个目标参数 `gate_up_proj`，再靠 `loaded_shard_id` 决定拷贝到 packed 参数的哪一段。

### 8.4 它和 QKVParallelLinear 的关系

两者本质上是同类技巧：

- `QKVParallelLinear`：把 `q/k/v` 合成一次列并行投影
- `MergedColumnParallelLinear`：把 `gate/up` 合成一次列并行投影

目标都一样：

> **减少 launch / 减少多次独立 matmul 的开销。**

## 九、词表并行：Embedding 和 LM Head 为什么长得不一样

源码路径：`nanovllm/layers/embed_head.py`

这一节很重要，因为它最容易出现“我以为也是 all_reduce / all_gather，结果代码不是这么写”的混淆。

### 9.1 `VocabParallelEmbedding`：切的是词表行

当前构造：

```python
self.num_embeddings_per_partition = self.num_embeddings // self.tp_size
self.vocab_start_idx = self.num_embeddings_per_partition * self.tp_rank
self.vocab_end_idx = self.vocab_start_idx + self.num_embeddings_per_partition
self.weight = nn.Parameter(torch.empty(self.num_embeddings_per_partition, embedding_dim))
```

这说明每个 rank 只保存词表中的一段：

```text
rank 0: [0, vocab/TP)
rank 1: [vocab/TP, 2*vocab/TP)
...
```

### 9.2 为什么 Embedding 要 `all_reduce`

前向实现是：

```python
if self.tp_size > 1:
    mask = (x >= self.vocab_start_idx) & (x < self.vocab_end_idx)
    x = mask * (x - self.vocab_start_idx)
y = F.embedding(x, self.weight)
if self.tp_size > 1:
    y = mask.unsqueeze(1) * y
    dist.all_reduce(y)
return y
```

它的含义是：

1. 每个 rank 先判断哪些 token 属于自己的词表分片；
2. 属于自己的 token，就转换成本地词表下标；
3. 不属于自己的 token，后面会被 `mask` 清零；
4. 每个 rank 算出“自己负责的那部分 embedding”；
5. 再 `all_reduce(sum)`。

为什么这里要 `all_reduce`？

因为 embedding 之后，后面的 Transformer 层要求：

> **所有 rank 都持有同样的完整 hidden_states，才能继续做后续层。**

所以 embedding 输出必须在每个 rank 上都变成一致的完整结果。

### 9.3 `ParallelLMHead`：为什么当前实现不是 `all_gather` 到所有 rank

先看当前代码：

```python
logits = F.linear(x, self.weight)
if self.tp_size > 1:
    all_logits = [torch.empty_like(logits) for _ in range(self.tp_size)] if self.tp_rank == 0 else None
    dist.gather(logits, all_logits, 0)
    logits = torch.cat(all_logits, -1) if self.tp_rank == 0 else None
return logits
```

这说明：

- 每个 rank 先算自己那部分词表 logits
- 然后不是所有 rank 都拿完整 logits
- 而是只把各自结果 `gather` 到 rank 0
- 最后只有 rank 0 `torch.cat(...)` 成完整词表 logits

这和很多通用教材里的“all_gather 到所有卡”不完全一样。

原因也很直接：

> **nano-vllm 最终只需要 rank 0 采样，所以没必要让所有 rank 都持有完整 logits。**

这是一种很合理的通信优化。

### 9.4 为什么 Embedding 是 `all_reduce`，LM Head 是 `gather`

这是非常值得面试直接讲的一组对照：

| 模块 | 目标 | 当前通信 |
|------|------|---------|
| `VocabParallelEmbedding` | 让所有 rank 继续拿同样的完整 hidden state 往下算 | `all_reduce` |
| `ParallelLMHead` | 只让 rank 0 拿到完整 logits 进行采样 | `gather(dst=0)` |

也就是说，关键不是“它们都和词表有关”，而是：

> **后续到底要求“所有 rank 都继续用这份结果”，还是“只有 rank 0 最终消费这份结果”。**

### 9.5 一个细节：prefill 时 LM Head 只取每条序列最后位置

当前 `ParallelLMHead.forward()` 里还有一句：

```python
if context.is_prefill:
    last_indices = context.cu_seqlens_q[1:] - 1
    x = x[last_indices].contiguous()
```

这和课程 14 对上了：

- prefill 阶段虽然模型内部可能处理了很多 token
- 但最终用于采样的，只是每条序列本轮 Q 片段的最后一个位置

这句和 TP 没直接冲突，但如果你把课程 14、15 一起串起来理解，会更稳。

## 十、当前 nano-vllm 里，哪些层在 TP，哪些层没有 TP

如果你第一次看源码，很容易误以为“整个模型每个模块都被 TP 切了”。实际上不是。

### 10.1 明确使用 TP 的地方

| 位置 | 实现 |
|------|------|
| token embedding | `VocabParallelEmbedding` |
| QKV 投影 | `QKVParallelLinear` |
| attention 输出投影 | `RowParallelLinear` |
| FFN gate/up | `MergedColumnParallelLinear` |
| FFN down | `RowParallelLinear` |
| LM Head | `ParallelLMHead` |

### 10.2 不需要 TP 切分的典型地方

例如：

- `RMSNorm`
- `RoPE`
- attention kernel 本体
- 激活函数
- 残差连接

这些模块要么参数量不大，要么天然能在本地 shard 上做，不需要额外定义复杂的 TP 参数布局。

要特别注意：

> **attention kernel 本体并不是“没并行”，而是“头已经先在 `qkv_proj` 阶段分给各 rank 了，所以 kernel 直接在各 rank 本地头上算”。**

## 十一、通信地图：当前实现里到底哪些地方在做 NCCL

如果你想真正把 TP 看清楚，不要只记概念，最好直接记“哪些函数里真的调用了分布式通信原语”。

### 11.1 `RowParallelLinear.forward()`

```python
dist.all_reduce(y)
```

用途：

- attention 的 `o_proj`
- FFN 的 `down_proj`

结果：

- 所有 rank 都拿到相同的完整输出

### 11.2 `VocabParallelEmbedding.forward()`

```python
dist.all_reduce(y)
```

用途：

- 把各 rank 负责的局部词表 embedding 合成完整 hidden state

结果：

- 所有 rank 都拿到相同的 embedding 输出

### 11.3 `ParallelLMHead.forward()`

```python
dist.gather(logits, all_logits, 0)
```

用途：

- 把局部词表 logits 收集到 rank 0

结果：

- 只有 rank 0 最终拼出完整 logits

### 11.4 为什么每层通常说“只有 2 次通信”

这里说的是**每个 Transformer block 内部**最主要的 TP 通信：

1. attention 的 `o_proj`
2. FFN 的 `down_proj`

Embedding 和 LM Head 分别位于：

- 模型最前面
- 模型最后面

所以通常讨论“每层通信量”时，不把它们算进每个 block 的循环里。

## 十二、把一层数据流完整走一遍

假设：

```text
tp_size = 2
hidden_size = 4096
num_heads = 32
num_kv_heads = 4
intermediate_size = 14336
```

则：

- 每 rank 16 个 Q 头
- 每 rank 2 个 KV 头
- 每 rank 负责一半中间维度

这一层可以按下面的方式读：

```text
输入 x: [M, 4096]，所有 rank 一样

1. qkv_proj（列并行）
   rank 0 -> 本地 Q/K/V shard
   rank 1 -> 本地 Q/K/V shard

2. attention（本地头）
   rank 0 只算自己的头
   rank 1 只算自己的头

3. o_proj（行并行）
   两边各自得到 [M, 4096] 的 partial sum
   all_reduce 后，两边都拿到相同完整 attention 输出

4. gate_up_proj（列并行）
   两边各算一半 FFN 中间维度

5. silu + mul（本地）
   各自只在自己的 shard 上算

6. down_proj（行并行）
   两边各自得到 [M, 4096] 的 partial sum
   all_reduce 后，两边都拿到相同完整 FFN 输出
```

这条数据流里最应该记的不是数字，而是节奏：

> **完整输入 -> 切出局部特征 -> 局部算子继续消费 -> 需要完整结果时再通信合并。**

## 十三、面试高频考点

### Q1：张量并行和数据并行的本质区别是什么？

**标准回答：**

数据并行是“每张卡放完整模型，处理不同 batch 样本”；张量并行是“同一个样本、同一个层，由多张卡共同持有不同权重分片并协同完成一次前向”。所以 TP 切的是层内权重和激活，DP 切的是 batch 样本。

### Q2：`ColumnParallelLinear` 和 `RowParallelLinear` 的核心区别是什么？

**标准回答：**

`ColumnParallelLinear` 让所有 rank 看到完整输入，但每个 rank 只负责输出维度中的一部分，因此输出是分片结果，通常不需要立刻通信；`RowParallelLinear` 让每个 rank 只看到输入分片，但各自会算出完整输出 shape 的 partial sum，因此必须 `all_reduce(sum)` 才能恢复真正完整输出。简短地说：Column 是“分输出”，Row 是“分输入再求和”。

### Q3：为什么 `ColumnParallelLinear` 在代码里是 `tp_dim=0`，这不是和“按列切分”矛盾吗？

**标准回答：**

不矛盾。数学上把线性层写成 `Y = XW` 时，`W` 形状通常记为 `[K, N]`，按输出维度切分就是按 `W` 的列切；但 PyTorch `F.linear` 的参数 `weight` 存的是 `[N, K]`，也就是 `[output, input]`。所以数学上的“按列切分输出维度”，落到 PyTorch 参数张量上对应的是第 0 维切分，这就是 `tp_dim=0` 的来源。

### Q4：为什么 Transformer 里总是“ColumnParallel -> 本地算子 -> RowParallel + AllReduce”？

**标准回答：**

因为先按输出维度切开后，每个 rank 能独立算出自己那部分特征，而中间很多算子并不需要跨 shard 混合信息，例如每个头独立 attention、RoPE、激活函数、逐元素乘等，都可以本地完成。只有当下一步真的需要完整 hidden state 时，才通过 `RowParallelLinear` 产出完整 shape 的 partial sum，再用 `all_reduce` 合成完整输出。这能最大化推迟通信、减少同步开销。

### Q5：为什么 `RowParallelLinear` 里 bias 只在 rank 0 添加？

**标准回答：**

因为 RowParallel 最后会 `all_reduce(sum)`。如果每个 rank 都加一遍 bias，那么最终结果会多出 `tp_size` 倍 bias。只让一个 rank 加 bias，再随 `all_reduce` 一起求和，最终就只会保留一份 bias。

### Q6：为什么 `VocabParallelEmbedding` 用 `all_reduce`，而 `ParallelLMHead` 在当前 nano-vllm 里是 `gather` 到 rank 0？

**标准回答：**

因为 embedding 输出之后，所有 rank 都还要继续参与后续层计算，所以每个 rank 都必须拿到同样的完整 hidden state，因此用 `all_reduce`。而 LM Head 之后，当前 nano-vllm 只有 rank 0 需要完整 logits 来采样，所以各 rank 先算本地词表 logits，再 `gather` 到 rank 0 即可，没必要让所有 rank 都持有完整 logits。

### Q7：GQA 为什么会限制 `tp_size`？

**标准回答：**

因为 TP 需要把 Q 头和 KV 头都均匀分到各 rank。当前实现要求：

```python
assert total_num_heads % tp_size == 0
assert total_num_kv_heads % tp_size == 0
```

例如总 Q 头数 32、总 KV 头数 4 时，`tp_size` 可以是 1、2、4，但不能是 8，因为 4 个 KV 头没法均匀分给 8 张卡。

### Q8：当前 nano-vllm 一个 Transformer block 里最主要的 TP 通信有哪些？

**标准回答：**

每个 block 内主要有 2 次 `all_reduce`：

1. attention 的 `o_proj(RowParallelLinear)`
2. FFN 的 `down_proj(RowParallelLinear)`

Embedding 前面还有一次 `all_reduce`，LM Head 最后还有一次 `gather` 到 rank 0，但它们不在每个 block 的重复路径里。

## 十四、小结

这课真正要记住的，不是零碎公式，而是下面这条主线：

```text
TP 不是拆请求，而是拆层里的大矩阵
  -> ColumnParallel 先把输出维度切开
  -> 中间一串不需要跨 shard 混合的算子都本地执行
  -> RowParallel 再把局部结果变成完整 shape 的 partial sum
  -> all_reduce 后恢复完整 hidden state
```

再把几个最关键的判断句记住：

- `ColumnParallel`：完整输入，分片输出，不急着通信
- `RowParallel`：分片输入，完整 shape 的部分和，必须 `all_reduce`
- `Embedding`：后续所有 rank 都要继续用，所以 `all_reduce`
- `LM Head`：当前只有 rank 0 采样，所以 `gather` 到 rank 0

如果要把张量并行浓缩成一句面试回答，可以直接说：

> **nano-vllm 的张量并行，本质是在单层内部按输出维度或输入维度切分权重，让各 rank 尽可能长时间只做本地 shard 计算，只有在必须恢复完整 hidden state 时才通过 NCCL 通信合并结果。**

最后给你一个记忆口诀：

> 不是拆请求，是拆矩阵；  
> Column 分输出，Row 分输入；  
> Column 产局部分片，Row 产完整 shape 的部分和；  
> attention 和 FFN 都是“先列后行”；  
> embedding 要全员拿全量，lm head 只把 logits 交给 rank 0。
