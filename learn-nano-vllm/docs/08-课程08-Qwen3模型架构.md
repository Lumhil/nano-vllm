# 课程08：Qwen3 模型架构

> 这节课最适合按“整机数据流”来读。先看懂 `input_ids` 进来后一路经过哪些模块，再分别拆 `DecoderLayer`、`Attention`、`MLP`、`GQA` 和权重映射，整篇就不会散。

## 本课目标

- 能按数据流口述 `Qwen3ForCausalLM -> Qwen3Model -> Qwen3DecoderLayer -> Qwen3Attention / Qwen3MLP` 的职责划分。
- 理解 Qwen3 为什么是标准 **Decoder-only** 架构，以及它和“编码器-解码器”模型的区别。
- 理解 **GQA（Grouped-Query Attention）**：为什么 `num_heads` 可以大于 `num_kv_heads`，以及它为什么能省 KV Cache。
- 看懂 `qkv_proj`、`gate_up_proj` 这些“融合层”名字，不再被 `packed_modules_mapping` 绕晕。
- 记住一个推理侧细节：`forward` 返回的是最后隐状态，不是 logits；logits 由 `compute_logits` 单独计算。

## 先认几个模块名

如果你第一次看 `nanovllm/models/qwen3.py`，建议先把这些名字当成“角色标签”来记：

- `Qwen3ForCausalLM`：最外层整机，里面包着模型主体和 `lm_head`
- `Qwen3Model`：真正的主干网络，负责 embedding、多层 decoder、最终 norm
- `Qwen3DecoderLayer`：模型里重复堆叠的“一层”
- `Qwen3Attention`：这一层里的注意力模块
- `Qwen3MLP`：这一层里的前馈网络
- `lm_head`：把最后隐状态映射成词表分数
- `GQA`：Q 头多，KV 头少；多个 Query 头共享同一组 K/V
- `TP`：张量并行，不同 GPU 各拿一部分权重和头
- `packed_modules_mapping`：把 HuggingFace 的“拆开权重”映射到 nano-vllm 的“融合权重”

你可以先不急着抠每个英文名的全称，只要先知道它在整条流水线里干什么。

## 先用一句人话概括 Qwen3

Qwen3 在这个仓库里的主体，其实就是一个很标准的 **Decoder-only Transformer**：

1. 输入 token id
2. 先查 embedding，变成向量
3. 重复很多层 `DecoderLayer`
4. 最后做一次 `RMSNorm`
5. 得到最后隐状态
6. 再用 `lm_head` 把隐状态投到词表上，得到 logits

也就是说，它的主线并不复杂。  
这篇代码真正容易让人卡住的地方，不是“模型太奇怪”，而是它在标准结构上又叠了几层工程实现：

- 用 **GQA** 降低 KV Cache 开销
- 用 **TP** 把头和权重切到多卡
- 用 **融合线性层** 减少 GEMM 次数
- 用 **单独的 `compute_logits`** 适配推理解码流程

所以一个很重要的阅读原则是：

**先抓主干，再看这些工程优化，不要一上来就钻进细节。**

## 先看整机数据流

整机最核心的路径可以先记成下面这样：

```text
input_ids
   |
   v
embed_tokens
   |
   v
Qwen3DecoderLayer x N
   |
   v
final RMSNorm
   |
   v
hidden_states
   |
   v
compute_logits / lm_head
   |
   v
logits
```

这里最值得先记住的一点是：

- `Qwen3ForCausalLM.forward(...)` 返回的是 `hidden_states`
- `compute_logits(hidden_states)` 才会调用 `lm_head` 算词表分数

这和很多“forward 直接给 logits”的入门代码不一样。

为什么要这么设计？

因为推理引擎在解码时，往往只关心：

- 当前步最后一个 token 的隐状态
- 什么时候去算 logits
- logits 之后怎么采样

把 `compute_logits` 单独拎出来，更利于和 scheduler、sampler、CUDA Graph 之类的推理组件解耦。

## 从外到内看：每个类负责什么

### 1. `Qwen3ForCausalLM`

它是最外层封装，主要做三件事：

- 持有 `self.model = Qwen3Model(config)`
- 持有 `self.lm_head = ParallelLMHead(...)`
- 如果 `tie_word_embeddings=True`，就让 `lm_head.weight` 和 `embed_tokens.weight` 共用同一份权重

可以把它理解成：

**“模型主干 + 输出头”的总装类。**

### 2. `Qwen3Model`

它是主干网络，负责把 token 一路变成最后隐状态：

```python
hidden_states = self.embed_tokens(input_ids)
residual = None
for layer in self.layers:
    hidden_states, residual = layer(positions, hidden_states, residual)
hidden_states, _ = self.norm(hidden_states, residual)
return hidden_states
```

这一段特别值得你读顺，因为它几乎就是模型结构本身：

- `embed_tokens`：把 `input_ids` 变成向量
- `self.layers`：重复跑很多层 decoder
- `self.norm`：最后再做一次归一化

一句话理解：

**`Qwen3Model` 就是在组织“embedding -> 多层 decoder -> final norm”这条主干。**

### 3. `Qwen3DecoderLayer`

每一层 decoder 都由两大块组成：

- `self_attn`：让 token 去看上下文
- `mlp`：让每个 token 自己再做一轮特征加工

它的 forward 主线是：

```python
if residual is None:
    hidden_states, residual = self.input_layernorm(hidden_states), hidden_states
else:
    hidden_states, residual = self.input_layernorm(hidden_states, residual)

hidden_states = self.self_attn(positions, hidden_states)
hidden_states, residual = self.post_attention_layernorm(hidden_states, residual)
hidden_states = self.mlp(hidden_states)
return hidden_states, residual
```

你可以把它翻译成人话：

1. 先整理输入，并维护残差流
2. 做注意力
3. 再次接回残差并归一化
4. 做前馈网络
5. 把新的主分支和残差分支交给下一层

如果你学过课程07，会发现这里沿用了同样的 Pre-Norm 残差写法：

- 第一层时，`residual` 还没有历史值
- 后续层时，`RMSNorm` 会同时处理主分支和残差分支

所以这层的核心，不是“奇技淫巧”，而是标准 Transformer layer 的工程实现版本。

## 单层里最重要的一块：Attention 到底怎么跑

### 先看最简流程

`Qwen3Attention.forward` 的主线非常清楚：

```python
qkv = self.qkv_proj(hidden_states)
q, k, v = qkv.split([self.q_size, self.kv_size, self.kv_size], dim=-1)
q = q.view(-1, self.num_heads, self.head_dim)
k = k.view(-1, self.num_kv_heads, self.head_dim)
v = v.view(-1, self.num_kv_heads, self.head_dim)
if not self.qkv_bias:
    q = self.q_norm(q)
    k = self.k_norm(k)
q, k = self.rotary_emb(positions, q, k)
o = self.attn(q, k, v)
output = self.o_proj(o.flatten(1, -1))
return output
```

翻成人话就是：

1. 先一次线性投影，直接得到混在一起的 Q/K/V
2. 再把它拆成 Q、K、V 三段
3. 改成多头形状
4. 某些配置下再对 Q、K 做一次 RMSNorm
5. 对 Q、K 加 RoPE 位置编码
6. 做注意力
7. 最后再投影回 `hidden_size`

### 为什么这里看起来“怪”，其实是因为做了融合

如果你看的是教科书版 Transformer，往往会写成：

```text
q = q_proj(x)
k = k_proj(x)
v = v_proj(x)
```

但这里不是三次线性层，而是：

```text
qkv = qkv_proj(x)
再 split 成 q / k / v
```

原因很简单：

- 一次大矩阵乘通常比三次小矩阵乘更高效
- 更容易吃满 GPU 的并行能力
- 权重布局更紧凑，工程上也更利于推理优化

所以别把 `qkv_proj` 理解成“多了一个神秘模块”，它本质上还是在做 Q/K/V 投影，只是把三步合并成了一步。

## GQA 到底是什么

### 1. 先别背定义，先看现象

在代码里：

```python
self.total_num_heads = num_heads
self.total_num_kv_heads = num_kv_heads
```

这意味着：

- Query 头数是 `num_heads`
- Key/Value 头数是 `num_kv_heads`

而这两个数不一定相等。

如果是最朴素的多头注意力（MHA），那就是：

```text
num_heads == num_kv_heads
```

但在 GQA 里，往往是：

```text
num_heads > num_kv_heads
```

也就是：

**很多个 Q 头，共享更少的 K/V 头。**

### 2. 一个直观例子

假设：

- `num_heads = 8`
- `num_kv_heads = 2`

那可以粗略理解成：

```text
Q0 Q1 Q2 Q3 共享较少的一组 KV
Q4 Q5 Q6 Q7 共享较少的一组 KV
```

更概念化一点，可以写成：

```text
Q:   q0 q1 q2 q3 q4 q5 q6 q7
KV:  kv0      kv1
```

具体哪几个 Q 头对应哪一个 KV 头，要看实现里的分组和广播方式；但阅读这一课时，你先抓住“Q 多、KV 少、KV 被共享”就够了。

### 3. 凭什么可以共享同一组 K/V

这是第一次接触 GQA 时最容易不服气的地方：

- 为什么 `Q` 可以有很多头
- 但 `K/V` 却可以更少
- 这样不会把注意力算坏吗

关键点在于：

**注意力公式要求的是“每个 Query 都能拿到一组可比较的 K，并用这组 K 对应的 V 做加权求和”，并没有强制规定“每个 Q 头必须拥有自己独立的一套 K/V 头”。**

先看最普通的多头注意力。第 `i` 个头通常可以写成：

```text
head_i = softmax(Q_i K_i^T / sqrt(d)) V_i
```

而在 GQA 里，可以把它改写成：

```text
head_i = softmax(Q_i K_g(i)^T / sqrt(d)) V_g(i)
```

这里的 `g(i)` 表示：

- 第 `i` 个 Query 头属于哪一个 KV 组
- 同一组里的多个 Query 头，共享同一组 `K` 和 `V`

所以 GQA 不是“把注意力公式改坏了”，而是把原来“一头一份 K/V”改成了“多头共享一份 K/V”。

更重要的是：

**即使多个 Q 头共享同一组 K/V，它们算出来的注意力结果也不一定一样。**

因为真正决定“关注哪里”的，不只是 `K`，还有每个头自己的 `Q`：

- `Q0` 和共享的 `K` 做点积，得到一套分数
- `Q1` 和同一份 `K` 做点积，仍然可能得到另一套分数
- 分数不同，softmax 后的权重就不同
- 权重不同，最后加权出来的输出也不同

你可以把它想成：

- `K/V` 像一份共享的资料库
- 不同的 `Q` 头像不同的提问方式

虽然大家都在查同一份资料库，但：

- 有的头更关心最近的 token
- 有的头更关心长距离依赖
- 有的头更关心语法结构

所以它们最后读出来的内容仍然可以不同。

一句话说，GQA 的核心假设是：

**查询视角可以很多样，但被查询的“记忆表示”不一定需要同样多份独立副本。**

这也是为什么工程上会觉得它合理：

- 保留较多的 Q 头，保住“怎么看”的多样性
- 压缩 K/V 头，减少“被存下来、被读取的记忆副本”

在本仓库代码里，这件事也能直接看出来。在 `Qwen3Attention.forward` 里：

```python
q = q.view(-1, self.num_heads, self.head_dim)
k = k.view(-1, self.num_kv_heads, self.head_dim)
v = v.view(-1, self.num_kv_heads, self.head_dim)
```

这三行就明确告诉你：

- `q` 的头数可以比 `k/v` 更多
- 后续 attention 内核会负责把多个 Q 头和较少的 KV 头按分组方式对应起来

所以“凭什么可以共享”的本质答案是：

**因为注意力本来就是“用 Query 去读 Key/Value 记忆”；多个 Query 完全可以读取同一份记忆，只要每个 Query 仍然保有自己的投影和自己的打分结果即可。**

### 4. 它为什么能省 KV Cache

因为在自回归推理里，最贵的缓存之一就是历史 token 的 K/V。

如果每个 Query 头都各存一份自己的 K/V，缓存就会比较大。  
而 GQA 把 K/V 头数减少后：

- K 的缓存更小
- V 的缓存更小
- 读写带宽压力也会下降

所以 GQA 的核心收益可以直接记成：

**牺牲一点点表达冗余，换来更省显存、更省带宽的长上下文推理。**

## 张量并行 TP 在这里怎么体现

在 `Qwen3Attention.__init__` 里有这段：

```python
tp_size = dist.get_world_size()
self.num_heads = self.total_num_heads // tp_size
self.num_kv_heads = self.total_num_kv_heads // tp_size
```

这表示：

- 总头数是配置里的全局头数
- 当前 rank 只拿到其中一部分头

你可以把它理解成：

**多卡时，不是每张卡都持有完整注意力头，而是每张卡只负责自己那一片。**

所以文档里经常看到“本地头数”“分片头数”，说的就是这个意思。

这里也解释了为什么代码要断言整除：

```python
assert self.total_num_heads % tp_size == 0
assert self.total_num_kv_heads % tp_size == 0
```

因为如果头数没法均匀切给每张卡，这种并行方式就没法成立。

## Q/K 上为什么还有一层 Norm

代码里有这样一个分支：

```python
if not self.qkv_bias:
    self.q_norm = RMSNorm(self.head_dim, eps=rms_norm_eps)
    self.k_norm = RMSNorm(self.head_dim, eps=rms_norm_eps)
```

前向里对应：

```python
if not self.qkv_bias:
    q = self.q_norm(q)
    k = self.k_norm(k)
```

这说明在某些配置下，Q 和 K 在进入 RoPE 和注意力之前，还会先做一次按 head_dim 的归一化。

阅读时你可以先这样理解：

- 这不是所有 Transformer 都有的“教科书默认步骤”
- 它更像是 Qwen3 配方里的一个具体实现细节
- 文档里只要记住“与 checkpoint 行为对齐”就够了

不用在这节课里把它神化成特别复杂的理论创新。

## MLP 这一块其实也没有表面上那么绕

### 1. 它的结构是什么

`Qwen3MLP` 的主线是：

```python
self.gate_up_proj = MergedColumnParallelLinear(
    hidden_size,
    [intermediate_size] * 2,
    bias=False,
)
self.down_proj = RowParallelLinear(
    intermediate_size,
    hidden_size,
    bias=False,
)
self.act_fn = SiluAndMul()
```

前向是：

```python
gate_up = self.gate_up_proj(x)
x = self.act_fn(gate_up)
x = self.down_proj(x)
```

### 2. 它在做什么

可以把它理解成一种 `SwiGLU` 风格的前馈层：

1. 先一次投影，得到两份中间结果
2. 一份当作 gate，一份当作 value
3. 用 `SiLU(gate) * value` 做门控
4. 再投影回原来的隐藏维度

所以 `gate_up_proj` 这个名字也就不神秘了：

- `gate` 那一半，对应门控分支
- `up` 那一半，对应上投影后的内容分支

它和前面的 `qkv_proj` 一样，本质也是：

**把原本拆开的几层合并起来，以换取更好的推理效率。**

## `packed_modules_mapping` 到底在解决什么问题

这部分是这篇文档里最容易把人绕进去的地方。  
先说结论：

**它解决的是“权重文件的命名方式”和“推理引擎内部模块组织方式”不一致的问题。**

HuggingFace 的权重名字，通常更“拆开”：

```text
q_proj
k_proj
v_proj
gate_proj
up_proj
```

但 nano-vllm 为了推理效率，会把它们融合成：

```text
qkv_proj
gate_up_proj
```

于是加载权重时，就需要一张“翻译表”：

```python
packed_modules_mapping = {
    "q_proj": ("qkv_proj", "q"),
    "k_proj": ("qkv_proj", "k"),
    "v_proj": ("qkv_proj", "v"),
    "gate_proj": ("gate_up_proj", 0),
    "up_proj": ("gate_up_proj", 1),
}
```

它的意思不是“模型结构变了”，而是：

- HF 文件里看到 `q_proj`
- 加载器就知道它其实应该塞进 `qkv_proj` 的 Q 那一段

同理：

- HF 文件里看到 `gate_proj`
- 加载器就知道它应该塞进 `gate_up_proj` 的第 0 段

所以你完全可以把 `packed_modules_mapping` 记成：

**“拆开权重 -> 融合权重”的对照表。**

## `tie_word_embeddings` 是什么

在最外层类里有这段：

```python
if config.tie_word_embeddings:
    self.lm_head.weight.data = self.model.embed_tokens.weight.data
```

它的含义是：

- 输入 embedding 用一张词表矩阵
- 输出 logits 的 `lm_head` 也用同一张矩阵

也就是常说的“输入输出权重共享”。

这么做的好处通常是：

- 参数更少
- 对某些模型来说效果也不错

所以这里不用想得太复杂，本质就是“同一张词表权重，入口和出口共用”。

### `Qwen3DecoderLayer.forward`

它定义了注意力主线：

- `qkv_proj`
- `split`

- `q_norm / k_norm`
- `rotary_emb`
- `attn`
- `o_proj`

### `Qwen3ForCausalLM`

它定义了推理接口层的两个关键点：

- `forward` 返回隐状态

## 面试考点怎么说更顺

### 1. Qwen3 在这里是什么架构

答法可以很直接：

> 它是标准的 Decoder-only Transformer。输入先经过 embedding，再经过多层 decoder layer，最后做 RMSNorm，并由 `lm_head` 计算 logits。

### 2. GQA 的核心收益是什么

> GQA 让 Query 头多于 KV 头，多个 Q 头共享同一组 K/V，因此 KV Cache 更小，长上下文推理更省显存和带宽。

### 3. 为什么 `forward` 不直接返回 logits

> 因为推理引擎常常把“主干网络算隐状态”和“算 logits、做采样”分成两个阶段，这样更便于调度和优化。

### 4. `packed_modules_mapping` 解决什么问题

> 它解决 HuggingFace 权重命名和 nano-vllm 融合模块命名不一致的问题，本质上是权重加载时的映射表。

### 5. 为什么要做 `qkv_proj` / `gate_up_proj` 这种融合

> 为了减少独立 GEMM 次数、改善内存局部性、提高 GPU 利用率，是典型的推理工程优化。

## 小结

Qwen3 这篇代码真正的主干并不复杂：

- `Qwen3Model` 负责整机数据流
- `Qwen3DecoderLayer` 负责单层结构
- `Qwen3Attention` 负责上下文交互
- `Qwen3MLP` 负责 token 内部特征加工

它之所以显得“难”，主要是因为在标准 Decoder-only 结构上叠加了几层推理工程实现：

- GQA
- 张量并行
- 融合线性层
- 拆开的 logits 计算
- 权重映射表

所以最好的阅读顺序永远是：

**先主线，后细节；先数据流，后优化项。**
