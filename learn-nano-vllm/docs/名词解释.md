## 参数和超参数有啥区别
- 参数 一般指模型内部的权重、偏置这些数值。比如大家常说“7B 参数模型”“70B 参数模型”，这里的“参数”就是模型在训练后得到的那些巨大数值矩阵。它们决定了模型“记住了什么、会怎么映射输入到输出”。
- 超参数 则不是模型学出来的，而是你在训练前或推理前手动指定的。它们不直接存储知识，而是控制“怎么学”或者“怎么生成”。比如：
> - 训练时的超参数：learning_rate、batch_size、epochs
> - 训练时的超参数：learning_rate、batch_size、epochs

## BPE 合并规则是啥
把训练语料里经常挨在一起出现的两个符号，合并成一个新的符号。这里的“符号”一开始通常是最小单位，比如字符，或者在很多大模型里更常见的是 byte。所以 BPE 叫 Byte Pair Encoding，可以理解成“字节对编码”。它不是语法规则，也不是人工写的语言学规则。它更像是一份按优先级排列的“拼接清单”：
- 哪两个相邻符号可以合并
- 谁优先级更高
- 合并到什么时候停止
通常你会在 tokenizer 文件里看到类似 merges.txt 的东西，那里面基本就是这些规则。
### 它怎么学出这些规则
训练 tokenizer 时，大概会这样做：
1. 先把文本拆成最小单位
2. 统计所有“相邻两项”出现的频率
3. 找出最常见的一对，比如 h + e
4. 把它合并成一个新符号 he
5. 重复很多轮
### 例子：
假设语料里经常出现 lower，训练后可能学到这些规则：
l + o -> lo
lo + w -> low
e + r -> er
low + er -> lower
那以后遇到 lower 这个词，tokenizer 就可能直接把它切成更少的 token，甚至一个 token。
`推理时怎么用这些规则？`:
当你输入一段文字时，tokenizer 会：
先拆成基础单位
按学好的 merge 规则不断合并
直到没有可用规则为止
所以 BPE 的效果就是：
常见片段会被合成更大的 token
生僻词也能退回到更小单位，不会完全“认不出来”
BPE 合并规则 = “把高频相邻片段不断合并”的历史记录和优先级表。

## logits是啥
logits 可以理解成：模型对每个候选 token 打的“原始分数”。从数学上说，logit 和对数几率有关，但在深度学习里，大家通常就把它当成softmax 前的原始输出分数，temperature 通常就是在 softmax 之前，去“拉伸/压缩” logits
它出现在生成的很关键一步里：
1. 你把一段文本喂给模型
2. 模型看完上下文
3. 对词表里的每个 token 都打一个分
4. 这一整排分数，就是 logits
5. 再把 logits 经过 softmax，变成概率
6. 再按 temperature、top_p 等规则去选下一个 token

## tokenizer
tokenizer_config.json   # tokenizer 配置:这个 tokenizer 使用时的外层配置
tokenizer.json          # tokenizer 定义:通常是 Hugging Face fast tokenizer 的完整序列化结果。怎么切词、怎么编码、怎么解码的完整规则
vocab.json              # 词表:有哪些 token，以及它们的编号
- 这三部分都属于 tokenizer 相关文件，作用不是“存模型知识”，而是把你的人类文本，变成模型能处理的 token id。
### tokenizer.json 
它回答的是：
- 输入文本先怎么预处理
- 怎么切分
- 用什么算法合并子词
- 特殊 token 怎么处理
- token ids 怎么再还原回文本

也就是说，它描述的是整条流水线：
```bash
原始文本
-> 规范化(normalize:比如要不要统一空格、大小写、Unicode 规范化)
-> 预切分(pre-tokenize:比如按空格切、按字节切、保留标点等)
-> 子词编码(具体采用什么 tokenizer 算法，比如 BPE / WordPiece / Unigram ...)
-> 得到 token ids
```
### tokenizer_config.json 是使用配置
这个 tokenizer 在实际推理时应该怎么用”的配置文件.它不一定决定切词算法本身，但会影响调用方式和默认行为
