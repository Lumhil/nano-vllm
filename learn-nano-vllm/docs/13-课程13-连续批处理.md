## 零、先讲人话版

如果你第一次读这课，最容易被“连续批处理”这个名字带偏。它不是说“把一堆请求永远塞进同一个大 batch 里连续跑”，而是：

> **每一步都重新看一眼：现在 GPU 上应该跑哪些序列？**

也就是说，连续批处理的“连续”，指的是：

1. **请求可以连续加入**：新请求不必等上一整个 batch 结束。
2. **请求可以连续退出**：某条序列一完成，资源立刻释放。
3. **batch 的成员连续变化**：这一步是 A/B/C，下一步可能就变成 B/C/D。

把 nano-vllm 的主循环压缩成人话，其实只有下面这条主线：

```text
有请求进来
  -> 先放进 waiting
  -> 每一轮 step 都让 scheduler 重新挑这轮参与者
  -> 这轮要么做 prefill，要么做 decode
  -> 做完后立刻更新状态
  -> 完成的退出，新的补进来
```

这里要特别先记住 3 个概念：

- **请求**：用户的一次生成任务。
- **Sequence**：请求在引擎里的运行时对象。
- **step / iteration**：引擎的一次调度-执行-后处理循环。

理解连续批处理的关键，不是先记一堆论文名，而是先盯住一句话：

> **连续批处理本质上是“step 级别重组 batch”，不是“请求级别锁死 batch”。**

如果你带着这个心智模型再去看静态批处理、动态批处理、Orca、vLLM、nano-vllm 的差别，后面会顺很多。

## 一、回顾：推理引擎为什么需要批处理

### 1.1 单请求推理的瓶颈

在 Decode 阶段，每一步只处理 **1 个 token**，但需要从显存中加载全部模型权重。以 7B 参数模型（FP16）为例：

- **模型权重大小**：\( 7 \times 10^9 \times 2 \text{ bytes} = 14 \text{ GB} \)
- **单 token 计算量**：约 \( 2 \times 7 \times 10^9 = 14 \text{ GFLOPs} \)
- **算术强度**：\( 14 \text{ GFLOPs} / 14 \text{ GB} = 1 \text{ FLOP/Byte} \)

A100 GPU 的峰值算力为 312 TFLOPS，显存带宽为 2 TB/s。对于算术强度为 1 的任务：

- **带宽瓶颈**：\( 2 \text{ TB/s} \times 1 \text{ FLOP/Byte} = 2 \text{ TFLOPS} \)
- **GPU 利用率**：\( 2 / 312 = 0.6\% \)

**GPU 99.4% 的算力都在闲置！** 这就是为什么必须通过批处理来提高利用率。

### 1.2 批处理的本质

批处理的核心思想：**将多个请求的计算合并到一次 GPU 操作中**。

当 batch_size = B 时：
- **数据搬运量不变**：仍然是 14 GB（模型权重只加载一次）
- **计算量变为 B 倍**：\( B \times 14 \text{ GFLOPs} \)
- **算术强度变为 B**：\( B \text{ FLOP/Byte} \)

当 B = 156 时，算术强度 = 156，GPU 利用率接近 100%。这就是批处理能够大幅提升吞吐量的数学原理。

## 二、静态批处理（Static Batching）

### 2.1 工作方式

静态批处理是最朴素的批处理策略：

1. 收集固定数量的请求组成一个 batch
2. 所有请求同时开始 Prefill
3. 所有请求同时进入 Decode
4. **必须等 batch 中最慢的请求完成**，整个 batch 才能结束
5. 结束后才能开始处理下一个 batch

```
时间 →
请求A: [Prefill][D1][D2][D3][D4][D5][D6][~~~~等待~~~~]
请求B: [Prefill][D1][D2][D3][D4][D5][D6][D7][D8][D9][D10]
请求C: [Prefill][D1][D2][D3][D4][~~~~等待~~~~~~~~~~~~~~~~~~~~~]
                                    ↑ C已完成但必须等B结束
```

### 2.2 静态批处理的问题

**问题 1：GPU 气泡（Bubble）**

当 batch 中的请求生成长度差异较大时，先完成的请求对应的 GPU 计算资源被浪费：

```
Batch 中 4 个请求的生成长度：[10, 50, 200, 500]
- 第 10 步后：只有 3 个请求在计算，1 个位置浪费
- 第 50 步后：只有 2 个请求在计算，2 个位置浪费
- 第 200 步后：只有 1 个请求在计算，3 个位置浪费
- 平均 GPU 利用率：(10×4 + 40×3 + 150×2 + 300×1) / (500×4) = 37%
```

**63% 的 GPU 算力被浪费！**

**问题 2：TTFT 延迟增大**

所有请求必须凑齐一个 batch 才开始处理。如果第一个请求到达后，需要等待其他请求到达才能组成 batch，TTFT 会显著增加。

**问题 3：吞吐量受限**

已完成的请求位置不能被新请求复用。整个 batch 的有效吞吐量取决于最慢的请求。

**问题 4：显存浪费**

必须为 batch 中每个请求按**最大可能生成长度**预分配 KV Cache 空间。实际生成长度通常远小于最大值，造成大量显存浪费。

### 2.3 静态批处理的适用场景

尽管有诸多问题，静态批处理在以下场景仍有价值：

- **离线批量推理**：所有请求已知，无需实时响应
- **固定长度生成**：如摘要生成（输出长度可控）
- **教学/原型验证**：实现简单，便于理解

## 三、动态批处理（Dynamic Batching）

### 3.1 改进思路

动态批处理在静态批处理的基础上做了一个关键改进：**允许在 batch 执行过程中加入新请求**。

```
时间 →
请求A: [Prefill][D1][D2][D3][完成]
请求B: [Prefill][D1][D2][D3][D4][D5][D6][D7][完成]
请求C:          [等待.....][Prefill][D1][D2][D3][完成]
                            ↑ A完成后，C加入batch
```

### 3.2 动态批处理的局限

虽然比静态批处理好，但动态批处理仍然存在问题：

1. **仍然是请求级调度**：新请求加入时，需要为其做 Prefill，但已有请求仍在 Decode。两种操作的计算特征完全不同，混合执行效率不高。
2. **Padding 浪费**：不同请求的输入长度不同，需要填充到相同长度。
3. **资源预留**：仍需为每个请求预留最大可能的 KV Cache 空间。

## 四、连续批处理（Continuous Batching）

### 4.1 核心思想

连续批处理（Continuous Batching）由 Orca 论文（Yu et al., 2022）提出，vLLM 和 nano-vllm 都实现了这一策略。其核心思想可以总结为三点：

1. **迭代级调度（Iteration-level Scheduling）**：每一次前向传播（iteration）都重新决定哪些序列参与计算
2. **即时加入与退出**：新请求可以在任意 iteration 加入，完成的请求立即退出
3. **Prefill 与 Decode 分离**：不同阶段的序列可以分开处理（或在 chunked-prefill 中混合处理）

```
时间步1: [A-prefill, B-prefill]       ← 两个新请求做 prefill
时间步2: [A-decode,  B-decode]         ← 两个请求进入 decode
时间步3: [C-prefill] 然后下一步 [A-decode, B-decode, C-decode] ← nano-vllm 单步 schedule 不混 prefill/decode；此处用两步表示“连续迭代”上的效果
时间步4: [A-完成,    B-decode, C-decode]  ← A 完成，立即释放资源
时间步5: [D-prefill]；时间步6: [B-decode, C-decode, D-decode]  ← D 在独立一步做 prefill，再与 B/C 同批 decode
```

### 4.2 与静态批处理的本质区别

| 特性 | 静态批处理 | 连续批处理 |
|------|-----------|-----------|
| 调度粒度 | 请求级（Request-level） | 迭代级（Iteration-level） |
| batch 生命周期 | 固定，从开始到所有请求完成 | 动态，每一步都可能变化 |
| 新请求加入 | 必须等当前 batch 结束 | 可以在任意 iteration 加入 |
| 请求退出 | 必须等所有请求结束 | 完成即退出，资源立即释放 |
| GPU 利用率 | 低（有气泡） | 高（始终填满） |
| KV Cache 管理 | 预分配，浪费大 | 按需分配（PagedAttention） |

### 4.3 Token 级调度 vs 请求级调度

**请求级调度（Request-level Scheduling）**：

以"请求"为单位做调度决策。一旦一个请求开始被处理，就持续处理直到完成。

**Token 级调度（Token-level Scheduling）**：

以"token"为单位做调度决策。每生成一个 token 后，调度器都会重新评估所有序列，决定下一步处理哪些。

nano-vllm 的连续批处理实际上是**迭代级调度**——每次 `schedule()` 调用都重新决定参与本步计算的序列集合。这比请求级调度更灵活，但不是严格的 token 级调度（因为一步中所有序列共享相同的前向传播）。

```python
# nano-vllm 的主循环：每步都调用 schedule()
def step(self):
    seqs, is_prefill = self.scheduler.schedule()   # ← 每步重新调度
    num_tokens = sum(seq.num_scheduled_tokens for seq in seqs) if is_prefill else -len(seqs)
    token_ids = self.model_runner.call("run", seqs, is_prefill)
    self.scheduler.postprocess(seqs, token_ids, is_prefill)
```

### 4.4 连续批处理如何解决静态批处理的问题

| 静态批处理的问题 | 连续批处理的解决方案 |
|----------------|-------------------|
| GPU 气泡 | 请求完成后立即退出，位置被新请求填入 |
| TTFT 延迟大 | 新请求可以在下一个 iteration 立即做 prefill |
| 吞吐量受限 | GPU 始终保持满载，有效计算比例高 |
| 显存浪费 | 配合 PagedAttention 按需分配 KV Cache |

## 五、nano-vllm 中的连续批处理实现

### 5.1 架构总览

nano-vllm 的连续批处理由以下组件协作实现：

```
LLMEngine.step()          ← 驱动主循环
    ↓
Scheduler.schedule()      ← 迭代级调度决策
    ├── prefill 分支       ← 优先处理 waiting 队列中的新请求
    └── decode 分支        ← waiting 为空时处理 running 队列
    ↓
ModelRunner.run()         ← 执行前向传播
    ↓
Scheduler.postprocess()   ← 更新缓存、必要时追加 token、清理已完成序列
```

### 5.1.1 `LLMEngine.generate` / `step`：核心循环（教学精简版）

下面去掉 tqdm、吞吐统计等细节，只保留**连续批处理**最关心的控制流（与 `nano-vllm-main/nanovllm/engine/llm_engine.py` 逻辑一致，仅作阅读版）：

```python
def generate(self, prompts, sampling_params, use_tqdm=True):
    for prompt, sp in zip(prompts, sampling_params):
        self.add_request(prompt, sp)
    outputs = {}
    while not self.is_finished():
        output, num_tokens = self.step()
        for seq_id, token_ids in output:
            outputs[seq_id] = token_ids
    outputs = [outputs[seq_id] for seq_id in sorted(outputs.keys())]
    outputs = [{"text": self.tokenizer.decode(token_ids), "token_ids": token_ids} for token_ids in outputs]
    return outputs

def step(self):
    seqs, is_prefill = self.scheduler.schedule()
    num_tokens = sum(seq.num_scheduled_tokens for seq in seqs) if is_prefill else -len(seqs)
    token_ids = self.model_runner.call("run", seqs, is_prefill)
    self.scheduler.postprocess(seqs, token_ids, is_prefill)
    outputs = [(seq.seq_id, seq.completion_token_ids) for seq in seqs if seq.is_finished]
    return outputs, num_tokens
```

**要点**：

- 外层 `while not self.is_finished()`：每轮是一次**迭代**；是否 prefill、批里有哪些序列，全由 `schedule()` 决定。
- `step()` 内**先调度、再推理、再后处理**，顺序不能打乱：后处理里才会 `append_token` 并更新 `running`。
- 仓库中 `num_tokens` 在 prefill 步为各序列 `num_scheduled_tokens` 之和，decode 步为 `-len(seqs)`，仅供进度条估算吞吐；面试答「连续批处理」时重点讲 `schedule()` 即可。

### 5.2 调度逻辑与连续批处理的对应

#### Prefill 阶段：新请求加入批次

```python
# scheduler.py - schedule() 的 prefill 分支（当前仓库）
while self.waiting and len(scheduled_seqs) < self.max_num_seqs:
    seq = self.waiting[0]
    remaining = self.max_num_batched_tokens - num_batched_tokens
    if remaining == 0:
        break
    if not seq.block_table:
        num_cached_blocks = self.block_manager.can_allocate(seq)
        if num_cached_blocks == -1:
            break
        num_tokens = seq.num_tokens - num_cached_blocks * self.block_size
    else:
        num_tokens = seq.num_tokens - seq.num_cached_tokens
    if remaining < num_tokens and scheduled_seqs:
        break
    if not seq.block_table:
        self.block_manager.allocate(seq, num_cached_blocks)
    seq.num_scheduled_tokens = min(num_tokens, remaining)
    num_batched_tokens += seq.num_scheduled_tokens
    if seq.num_cached_tokens + seq.num_scheduled_tokens == seq.num_tokens:
        seq.status = SequenceStatus.RUNNING
        self.waiting.popleft()
        self.running.append(seq)
    scheduled_seqs.append(seq)
if scheduled_seqs:
    return scheduled_seqs, True   # ← prefill 模式
```

这段代码实现了连续批处理的**"即时加入"**特性：
- 每次 `schedule()` 调用，只要 waiting 非空且资源允许，就立即处理新请求
- 新请求不需要等待当前 batch 结束
- 多个新请求可以在同一步中一起做 prefill

但这里要补一个**很关键的细节**：在当前 nano-vllm 里，某条序列即使已经开始做 prefill，也**不一定马上从 waiting 进入 running**。如果它只是做了**部分 prefill**，那么它虽然已经参与了这轮计算，仍然会留在 waiting 里，下一轮继续补剩余上下文。

所以更准确地说：

- `waiting` 里不只有“全新请求”
- 还可能有“已经做过一部分 prefill、但还没准备好进入 decode”的序列

这也是连续批处理里很容易忽略的一点：**一个 step 被调度到，不等于这条序列马上进入 decode。**

#### Decode 阶段：已有请求继续生成

```python
# scheduler.py - schedule() 的 decode 分支（与仓库一致）
while self.running and len(scheduled_seqs) < self.max_num_seqs:
    seq = self.running.popleft()
    while not self.block_manager.can_append(seq):
        if self.running:
            self.preempt(self.running.pop())
        else:
            self.preempt(seq)
            break
    else:
        seq.num_scheduled_tokens = 1
        seq.is_prefill = False
        self.block_manager.may_append(seq)
        scheduled_seqs.append(seq)
assert scheduled_seqs
self.running.extendleft(reversed(scheduled_seqs))
return scheduled_seqs, False   # ← decode 模式
```

这段代码实现了连续批处理的**"动态 batch 组合"**特性：
- 每步 decode 时，从 running 队列中选取序列组成 batch
- 如果资源不足，通过抢占释放 block
- batch 的组成在每一步都可能不同

#### 后处理：完成请求立即退出

```python
# scheduler.py - postprocess()
def postprocess(self, seqs, token_ids, is_prefill):
    for seq, token_id in zip(seqs, token_ids):
        self.block_manager.hash_blocks(seq)
        seq.num_cached_tokens += seq.num_scheduled_tokens
        seq.num_scheduled_tokens = 0
        if is_prefill and seq.num_cached_tokens < seq.num_tokens:
            continue
        seq.append_token(token_id)
        if (not seq.ignore_eos and token_id == self.eos) or seq.num_completion_tokens == seq.max_tokens:
            seq.status = SequenceStatus.FINISHED
            self.block_manager.deallocate(seq)   # ← 立即释放 KV Cache
            self.running.remove(seq)              # ← 立即从 batch 中退出
```

这段代码实现了连续批处理的**"即时退出"**特性：
- 一旦序列生成了 EOS 或达到 max_tokens，**立即**释放资源
- 不需要等待 batch 中其他序列完成
- 释放的 block 在下一步就可以被新请求使用

同时也体现了另一个关键事实：

- 如果这一轮只是**部分 prefill**，`postprocess()` 会先推进缓存进度，然后直接 `continue`
- 也就是说，这一轮**不会立刻生成新 token**

所以连续批处理里的“一个 step”并不总是“生成一个 token 的一步”。对 prefill 来说，它也可能只是“继续把上下文铺进 KV Cache 的一步”。

### 5.3 Prefill 优先策略的含义

nano-vllm 采用 **Prefill 优先**策略，这意味着：

```
Step 1: waiting=[A,B,C], running=[]
        → prefill A,B （资源允许两个）
        → running=[A,B], waiting=[C]

Step 2: waiting=[C], running=[A,B]
        → prefill C （waiting 非空，优先 prefill）
        → running=[A,B,C], waiting=[]

Step 3: waiting=[], running=[A,B,C]
        → decode A,B,C
        → A 完成 → running=[B,C]

Step 4: waiting=[D]（新请求到达）, running=[B,C]
        → prefill D （waiting 非空，优先 prefill）
        → running=[B,C,D]

Step 5: waiting=[], running=[B,C,D]
        → decode B,C,D
```

上面这张图为了便于理解，默认假设这些请求的 prefill 都能在各自那一步里完整做完，所以会立即进入 `running`。真实实现里，如果 prompt 很长，也可能出现“这轮只做了部分 prefill，序列还留在 waiting”的情况。

注意 Step 4：即使 B、C 正在 decode，一旦新请求 D 到来，调度器会优先处理 D 的 prefill，B 和 C 本步不会 decode。这就是"Prefill 优先"的含义——**新请求的 TTFT 被最小化，但代价是已有请求的 TPOT 可能增加**。

再补一句和当前实现强相关的话：**nano-vllm 的“连续批处理”不等于“同一步里混合 prefill 和 decode”**。它的连续性体现在“step 与 step 之间 batch 成员持续变化”，而不是“同一个 step 里两种阶段混跑”。

### 5.4 nano-vllm 如何避免 Padding

传统批处理需要将不同长度的序列填充到相同长度，造成计算浪费。nano-vllm 通过以下技术避免 Padding：

**Prefill 阶段**：使用 `flash_attn_varlen_func`

```python
# model_runner.py - prepare_prefill()
input_ids, positions, cu_seqlens_q, cu_seqlens_k = [], [], [0], [0]
for seq in seqs:
    start = seq.num_cached_tokens
    seqlen_q = seq.num_scheduled_tokens
    end = start + seqlen_q
    seqlen_k = end
    input_ids.extend(seq[start:end])
    positions.extend(range(start, end))
    cu_seqlens_q.append(cu_seqlens_q[-1] + seqlen_q)
    cu_seqlens_k.append(cu_seqlens_k[-1] + seqlen_k)
```

所有序列的 token **拼接**成一个一维张量，通过 `cu_seqlens`（cumulative sequence lengths）记录每个序列的边界。FlashAttention 的变长接口能够处理这种格式，无需 Padding。这句话要注意歧义：FlashAttention 的变长接口为避免 Padding提供了帮助

这里也能看出连续批处理和部分 prefill 是怎么接上的：

- `start = num_cached_tokens`：跳过已经有 KV Cache 的那部分
- `seqlen_q = num_scheduled_tokens`：只处理这一轮真正被调度到的 token 数
- 所以 prefill 并不一定每次都把整条 prompt 一口气吃完

**Decode 阶段**：每个序列只有 1 个 token

```python
# model_runner.py - prepare_decode()
for seq in seqs:
    input_ids.append(seq.last_token)
    positions.append(len(seq) - 1)
```

所有序列各取 1 个 token，天然等长，无需 Padding。

## 六、GPU 利用率分析

### 6.1 更直观的理解：为什么连续批处理通常更划算

这一节如果硬上一个“手算总步数”的玩具公式，很容易把自己绕进去。更实用的理解方式其实是看**slot 占用率**。

假设 GPU 一次最多能服务 4 条 decode 序列：

```
静态批处理：
Step 1-10:  [A, B, C, D]
Step 11:    C 完成，但这个位置不能给新请求
Step 12-50: [A, B, -, D]
Step 51:    B 完成
Step 52-80: [A, -, -, D]
...

连续批处理：
Step 1-10:  [A, B, C, D]
Step 11:    C 完成，E 立刻补进来
Step 12-50: [A, B, E, D]
Step 51:    B 完成，F 立刻补进来
Step 52-80: [A, F, E, D]
...
```

连续批处理的收益，本质上来自这件事：

> **谁先完成，谁就立刻退出；空出来的位置马上给后续请求。**

所以它的优势通常体现在：

1. **slot 空转更少**：不需要等同一批里最慢的请求收尾。
2. **TTFT 更低**：新请求可以更早开始 prefill，不用等整批结束。
3. **在线场景更自然**：请求是陆续到达的，连续批处理更贴近真实服务流量。
4. **显存利用更细粒度**：配合 PagedAttention，释放和复用都更及时。

如果一定要用一句面试话术总结这一节，可以说：

> 连续批处理不一定让“每一步都更快”，但它通常让“整台 GPU 的有效工作比例更高”。

### 6.2 实际 GPU 利用率的影响因素

| 因素 | 影响 | nano-vllm 中的参数 |
|------|------|-------------------|
| max_num_seqs | batch 越大，GPU 算术强度越高 | `config.max_num_seqs` |
| max_num_batched_tokens | 限制单步 prefill 的 token 总量 | `config.max_num_batched_tokens` |
| KV Cache block 总数 | 决定能同时服务多少序列 | `config.num_kvcache_blocks` |
| 请求到达速率 | 影响 waiting 队列长度和调度频率 | 外部输入 |
| 输出长度分布 | 影响序列退出/加入的频率 | 请求决定 |

### 6.3 Prefill 与 Decode 的资源竞争

在 nano-vllm 的 Prefill 优先策略下，Prefill 和 Decode 不会同时执行。这避免了两种操作的资源竞争，但引入了另一个问题：

```
时间 →
[Prefill A,B,C (800 tokens, 耗时较长)]
[Decode  D,E (2 tokens, 耗时很短)]
[Prefill F (200 tokens)]
[Decode  A,B,C,D,E (5 tokens)]
```

当 Prefill 处理长 prompt 时，Decode 被完全阻塞，已有序列无法产出新 token。用户感知到的现象是：**正在生成的回复突然"卡住"了**。

这就是 chunked prefill 的动机之一——将长 Prefill 拆分成多个小块，与 Decode 交替执行。

## 七、与 vLLM 和 Orca 的对比

### 7.1 Orca（OSDI 2022）

Orca 是连续批处理的开创性工作，其核心贡献：

1. **提出迭代级调度（Iteration-level Scheduling）**：每次前向传播都重新选择参与的序列
2. **选择性批处理（Selective Batching）**：在 Attention 层单独处理每个序列，只在 FFN 层做批处理
3. **支持 Prefill 和 Decode 混合**：同一个 batch 中可以同时有 prefill 序列和 decode 序列

nano-vllm 相对于 Orca 的**简化**：
- 不在同一步中混合 prefill 和 decode
- 不做选择性批处理（使用 FlashAttention 统一处理）

### 7.2 vLLM（SOSP 2023）

vLLM 在 Orca 的基础上引入了 **PagedAttention** 和更完善的调度机制：

| 特性 | vLLM | nano-vllm |
|------|------|-----------|
| 连续批处理 | ✅ | ✅ |
| PagedAttention | ✅ | ✅ |
| Chunked Prefill | ✅ | ✅（轻量版，仅 prefill-only） |
| Swap（CPU↔GPU） | ✅ | ❌（用 recompute） |
| 前缀缓存 | ✅ | ✅ |
| Beam Search | ✅ | ❌ |
| SequenceGroup | ✅ | ❌ |
| Speculative Decoding | ✅ | ❌ |
| 多节点推理 | ✅ | ❌ |

### 7.3 Chunked Prefill

Chunked Prefill 是 vLLM 后续版本引入的重要优化方向，它解决了长 prefill 阻塞 decode 的问题。但这里要注意，**当前 nano-vllm 并不是完全没有 chunked prefill**，而是只有一个非常轻量的版本。

**问题**：长 prompt 的 Prefill 会阻塞 Decode。

**完整方案（vLLM）**：将长 Prefill 拆分为多个 chunk，每个 chunk 与 Decode batch 混合执行。

```
传统（nano-vllm）：
Step 1: [Prefill A, 1000 tokens]  ← 耗时长，Decode 被阻塞
Step 2: [Decode B, C, D, E]

Chunked Prefill（vLLM）：
Step 1: [Prefill A chunk1 (256 tokens), Decode B, C, D, E]
Step 2: [Prefill A chunk2 (256 tokens), Decode B, C, D, E]
Step 3: [Prefill A chunk3 (256 tokens), Decode B, C, D, E]
Step 4: [Prefill A chunk4 (232 tokens), Decode B, C, D, E]
```

这类完整 chunked prefill 的优点：
- Decode 不会被 Prefill 阻塞
- 更均匀的每步执行时间
- 更好的尾延迟（P99 latency）

这类完整 chunked prefill 的代价：
- 实现复杂度增加
- 注意力核需要支持混合模式
- Prefill 的总耗时可能增加（因为分多步处理）

**nano-vllm 当前的情况**：

- 支持一个**head-only、prefill-only** 的轻量 chunked prefill
- 也就是：如果 `waiting` 队头请求太长，它可以先切一部分 `num_scheduled_tokens` 出来跑，下一轮再续跑
- 但它**不会**把这个 prefill chunk 和 decode 放进同一个 step 里混跑

所以更准确地说：

- **vLLM 的 chunked prefill**：强调“chunk + decode 混合调度”
- **nano-vllm 的轻量 chunked prefill**：强调“长 prefill 可以分多轮续跑，但仍然与 decode 分步执行”

### 7.4 三者的调度策略对比

```
Orca:
  - 允许 prefill 和 decode 混合
  - 选择性批处理
  - 迭代级调度

vLLM:
  - 支持 chunked prefill（prefill + decode 混合）
  - PagedAttention 管理 KV Cache
  - Swap 机制处理 KV Cache 溢出
  - 完整的优先级和公平性保证

nano-vllm:
  - Prefill 优先，不做 prefill/decode 同步混合
  - 支持轻量的 prefill-only chunked prefill
  - PagedAttention 管理 KV Cache
  - Recompute 策略（不做 swap）
  - 简洁实现，适合教学
```

## 八、深入理解：连续批处理的关键设计决策

### 8.1 Prefill 与 Decode 是否应该混合

**不混合（nano-vllm 方案）**：

优点：
- 实现简单，`prepare_prefill` 和 `prepare_decode` 独立
- 单步内所有序列的计算模式一致，利于 CUDA Graph 优化
- 调度逻辑简单清晰

缺点：
- Prefill 阻塞 Decode，影响 TPOT 的稳定性
- 长 prompt 场景下 Decode 等待时间不可控

**混合（vLLM chunked-prefill 方案）**：

优点：
- Decode 不受 Prefill 阻塞
- 更稳定的 TPOT
- GPU 利用率更高

缺点：
- 注意力核需要支持混合长度输入
- CUDA Graph 不易优化（每步 batch 形状不同）
- 调度逻辑复杂

### 8.2 抢占策略的选择

当 KV Cache 空间不足时，需要决定抢占哪些序列：

| 策略 | 描述 | nano-vllm |
|------|------|-----------|
| LIFO | 抢占最晚加入的序列 | ✅ 使用此策略 |
| FIFO | 抢占最早加入的序列 | ❌ |
| LRU | 抢占最近最少使用的 | ❌ |
| 最短剩余时间优先 | 估算剩余生成长度，抢占最长的 | ❌ |
| 随机 | 随机选择抢占 | ❌ |

nano-vllm 选择 LIFO 的原因：最晚加入的序列通常已生成最少的 token，抢占后重新 prefill 的代价最小。

### 8.3 max_num_seqs 和 max_num_batched_tokens 的关系

这两个参数共同控制连续批处理的行为：

- `max_num_seqs`：限制 batch 中最多包含多少个序列。主要受限于 KV Cache 总量和 CUDA Graph 的 batch size 限制。
- `max_num_batched_tokens`：限制单步 prefill 中最多处理多少个 token。主要受限于 GPU 显存（激活值大小）和计算耗时（影响 TTFT）。

两者的协同作用：

```
场景1: max_num_seqs=256, max_num_batched_tokens=2048
  → Prefill 时，最多一次处理 2048 个 token（可能是 1 个长 prompt 或多个短 prompt）
  → Decode 时，最多 256 个序列同时 decode

场景2: max_num_seqs=32, max_num_batched_tokens=8192
  → Prefill 时，可以处理很长的 prompt（但 batch 中序列数不多）
  → Decode 时，最多 32 个序列
```

在 nano-vllm 中，这两个参数在 `Config` 中设置，并传递给 `Scheduler`：

```python
class Scheduler:
    def __init__(self, config):
        self.max_num_seqs = config.max_num_seqs
        self.max_num_batched_tokens = config.max_num_batched_tokens
```

## 九、连续批处理的性能优化方向

### 9.1 CUDA Graph 与连续批处理

CUDA Graph 可以大幅减少 GPU kernel launch 开销，但要求输入形状固定。在连续批处理中，每步的 batch size 可能不同，这与 CUDA Graph 的要求矛盾。

nano-vllm 的解决方案：**预先为多种 batch size 捕获 CUDA Graph**，运行时选择第一个能容纳当前 batch size 的图。

```python
# model_runner.py
def capture_cudagraph(self):
    self.graph_bs = [1, 2, 4, 8] + list(range(16, max_bs + 1, 16))
    self.graphs = {}
    for bs in reversed(self.graph_bs):
        graph = torch.cuda.CUDAGraph()
        ...
        self.graphs[bs] = graph
```

Decode 阶段的 batch size 通常在 1 到 `max_num_seqs` 之间变化。运行时会取：

```python
graph = self.graphs[next(x for x in self.graph_bs if x >= bs)]
```

也就是选择“**第一个不小于当前 batch size 的图**”。这能在保持 decode 低开销的同时，适配连续批处理里 batch 大小不断波动的现实。

### 9.2 前缀缓存与连续批处理

前缀缓存（Prefix Caching）在连续批处理中的作用更加重要：

- 多个请求共享相同的 system prompt → 前缀的 KV Cache 可以复用
- 被抢占的序列重新调度时 → 可能部分命中前缀缓存，减少重新 prefill 的计算量
- 减少 Prefill 阶段的耗时 → Decode 被阻塞的时间更短

### 9.3 投机解码（Speculative Decoding）

投机解码是另一种提高连续批处理效率的技术：

- 使用小模型快速生成多个候选 token
- 大模型一次性验证所有候选
- 正确的候选直接采纳，减少大模型的 Decode 步数

nano-vllm 未实现此技术，但在 vLLM 中已经支持。

## 十、面试高频考点

### Q1：请解释连续批处理（Continuous Batching）的原理，以及它相对于静态批处理的优势。

**参考答案**：

连续批处理是一种迭代级调度策略，每次前向传播都重新决定哪些序列参与计算。相比静态批处理：

1. **即时加入**：新请求不需要等当前 batch 结束就可以加入，降低 TTFT
2. **即时退出**：完成的请求立即释放资源，不存在 GPU 气泡
3. **动态 batch**：batch 的组成在每一步都可能变化，GPU 始终保持满载
4. **配合 PagedAttention**：按需分配 KV Cache，避免显存浪费

### Q2：nano-vllm 的连续批处理实现有哪些特点和局限？

**参考答案**：

特点：
- **Prefill 优先策略**：有 waiting 请求时优先做 prefill，最小化 TTFT
- **不混合 prefill 和 decode**：一步只做一种操作，简化实现
- **轻量 chunked prefill**：允许 waiting 队头长请求按 step 分段续跑
- **Recompute 策略**：被抢占的序列重新 prefill 而非 swap 到 CPU
- **前缀缓存**：减少重新 prefill 的计算量

局限：
- 不支持生产级的“prefill + decode 混合 chunked prefill”，长 prompt 仍可能阻塞 decode
- 不支持 swap，抢占代价较高
- Prefill 优先可能导致 decode 被饿死

### Q3：Prefill 优先和 Decode 优先各有什么优缺点？实际系统中如何选择？

**参考答案**：

| | Prefill 优先 | Decode 优先 |
|--|-------------|------------|
| TTFT | 低 | 高 |
| TPOT 稳定性 | 差（被 prefill 打断） | 好 |
| 吞吐量 | 取决于请求到达模式 | 取决于请求到达模式 |
| 适用场景 | 交互式对话（用户等首字） | 批量生成（追求吞吐） |

实际系统通常使用 chunked prefill 作为折中：将 Prefill 拆成小块与 Decode 混合执行，兼顾 TTFT 和 TPOT 稳定性。补一句更细的：nano-vllm 现在已经支持一个轻量版的“按 step 切分 prefill”，但还没有做到和 decode 同步混跑。

### Q4：连续批处理如何避免 padding 浪费？

**参考答案**：

两个技术：
1. **Prefill 阶段**：使用 FlashAttention 的变长接口（`flash_attn_varlen_func`），将所有序列的 token 拼接为一维张量，通过 `cu_seqlens` 记录边界。无需将短序列填充到最长序列的长度。
2. **Decode 阶段**：每个序列只有 1 个 token，天然等长，使用 `flash_attn_with_kvcache` 配合 block_table 做分页注意力。

### Q5：请比较 Orca、vLLM 和 nano-vllm 的连续批处理实现。

**参考答案**：

- **Orca**（开创者）：提出迭代级调度和选择性批处理。允许在同一步中混合 prefill 和 decode 序列，但需要在 Attention 层分开处理。
- **vLLM**：在 Orca 基础上引入 PagedAttention 管理 KV Cache，支持 chunked prefill 将长 prefill 拆分并与 decode 混合、swap 机制处理 KV Cache 溢出、前缀缓存等高级功能。
- **nano-vllm**：简化版实现，保留了连续批处理和 PagedAttention 的核心思想。采用 prefill 优先、不做 prefill/decode 同步混合的策略，使用 recompute 代替 swap，并带有一个轻量版的 prefill-only chunked prefill，代码简洁适合教学。

### Q6：什么是 Chunked Prefill？它解决了什么问题？

**参考答案**：

Chunked Prefill 的完整形态，是将长 prompt 的 Prefill 拆分为多个 chunk（如每 chunk 256 tokens），每个 chunk 与 Decode batch 混合执行。要注意区分：nano-vllm 当前已有轻量版的“分 chunk 续跑 prefill”，但还没有做到与 decode 同步混跑。

解决的问题：
1. **Decode 阻塞**：传统 Prefill 优先策略中，长 prompt 会长时间阻塞 Decode，已有序列无法产出新 token
2. **TPOT 不稳定**：Decode 被间歇性打断，用户感知到生成速度忽快忽慢
3. **尾延迟高**：长 Prefill 导致 P99 延迟大幅增加

代价是实现复杂度增加，注意力核需要支持混合长度输入。

### Q7：连续批处理中，如何决定 max_num_seqs 的最优值？

**参考答案**：

`max_num_seqs` 的最优值取决于多个因素：

1. **KV Cache 总量**：`num_kvcache_blocks` 除以每个序列平均需要的 block 数，得到上限
2. **GPU 算力**：batch 过大会使 Decode 从 memory-bound 变为 compute-bound，TPOT 增加
3. **CUDA Graph 限制**：需要为所有可能的 batch size 预捕获 graph
4. **延迟 SLA**：TPOT 必须满足服务级别协议

一般做法是通过 profiling 确定：从小到大增加 max_num_seqs，观察吞吐量和 TPOT，选择吞吐量饱和且 TPOT 满足 SLA 的点。

## 十一、小结

| 要点 | 内容 |
|------|------|
| **静态批处理** | 固定 batch，等最慢请求完成才能开始新 batch。有 GPU 气泡和 TTFT 问题 |
| **连续批处理** | 迭代级调度，请求可即时加入/退出，GPU 始终满载 |
| **nano-vllm 策略** | Prefill 优先 + 不混合 + Recompute 抢占 + 前缀缓存 |
| **避免 Padding** | Prefill 用 `cu_seqlens` 变长注意力，Decode 天然等长 |
| **核心对比** | Orca（开创）→ vLLM（PagedAttention + Chunked Prefill）→ nano-vllm（教学简化） |
| **性能关键** | `max_num_seqs` 和 `max_num_batched_tokens` 共同控制吞吐量和延迟 |

**核心记忆口诀**：

> 静态批处理**等最慢**，连续批处理**不停转**；
> Prefill 优先**首字快**，Decode 轮转**不间断**；
> CUDA Graph **固形状**，前缀缓存**省重算**。
